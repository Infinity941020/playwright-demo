# 面談用プロジェクト説明

## 30秒版

このプロジェクトは、PlaywrightとTypeScriptを使用して作成したECサイト向けのE2E自動テストです。  
ログイン、商品追加、カート確認、購入、ログアウトまでの一連のユーザー操作を自動化しています。  

設計面ではPage Object Modelを採用し、ログイン処理はfixtureで共通化しました。  
また、GitHub ActionsによるCIも導入し、push時に自動でテストが実行される構成にしています。

---

## 1分版

PlaywrightとTypeScriptを用いて、ECサイトを想定したE2E自動テストを構築したプロジェクトです。  

ログイン機能、カート機能、Checkout（購入）機能、ログアウト機能まで、ユーザー操作全体を対象にテストを実装しました。  

保守性を意識してPage Object Modelを採用し、InventoryPage、CartPage、CheckoutPageなど画面ごとに責務を分離しています。  

また、ログイン処理はfixture化し、各テストで共通利用できるようにすることで、実行速度とメンテナンス性を改善しました。  

さらにGitHub ActionsによるCIを導入し、push時に自動で全テストが実行される構成にしています。

---

## 2分版

このプロジェクトは、実務を意識したE2E自動テストの学習・実践用として作成しました。  
使用技術はPlaywright、TypeScript、GitHub Actionsです。  

対象はECサイトのデモアプリで、以下の機能をテスト対象にしています。  

- ログイン正常系 / 異常系  
- 商品一覧からのカート追加  
- カート内商品の確認 / 削除  
- Checkoutによる購入処理  
- ログアウト機能  

設計面では、Page Object Modelを採用し、画面操作ロジックを各クラスへ分離しました。  
そのため、UI変更時の修正影響を局所化しやすく、テストコードの可読性も高めています。  

また、ログイン処理はfixtureで共通化し、毎回ログイン操作を実行せずに済むよう構成しました。  
これにより、テスト実行速度の改善と前処理重複の削減を実現しています。  

Checkout機能では、正常系・異常系・キャンセル系に分けて観点整理を行い、異常系ではdata-driven testingで入力パターンを管理しています。  

さらに、GitHub ActionsによるCIを導入し、push時に自動でPlaywrightテストが実行されるようにしています。  

単にテストを書くのではなく、保守性・再利用性・継続運用まで意識して構築したプロジェクトです。

## このプロジェクトで学んだこと

- E2Eテスト設計の進め方  
- Page Object Modelによる責務分離  
- fixtureによる共通処理設計  
- data-driven testingによる保守性向上  
- GitHub Actionsを使ったCI運用  
- テストコードの可読性・保守性の重要性
